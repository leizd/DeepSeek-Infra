# DeepSeek Infra 4.3.1 — Lazy Runtime Continuity and Offline Upgrade Safety

[中文](../../README.md) / [English](../../README.en.md)
<!-- docs-language-switcher:end -->

4.3.1 hardens the demand-loaded Workspace introduced in 4.3.0. Lazy mounting now preserves write exclusion across Providers, every visible retry maps to one distinct module request, and offline navigation always prefers the current build while an unrefreshed old page may still read its exact hash chunks.

## What changed

- Memory barriers are shared by `QueryClient`. Root suggestion saves and lazy drawer remove/clear actions see the same local blockers, retain the originating lifecycle ID and survive drawer close/reopen before any MutationCache fallback is needed.
- Workspace Feature and Skills Runtime recovery state is independent. The initial chunk identity has one unique retry identity; if that request also fails, the UI offers **Refresh app** and **Close** instead of a fake third retry. Render and business errors skip module retry.
- `workspace-assets.json` now emits four disjoint layers: `core`, `offlinePrimary`, `recovery` and `routeOptional`.
- Service Workers read the current build Cache first and use the previous Cache only for exact hashed asset requests. Shell HTML, `workspace-assets.json`, un-hashed metadata and search-bearing requests never fall through to an older build.
- Primary Workspace assets warm through `requestIdleCallback` with a five-second timeout. Save-Data, slow-2g and 2g connections skip warmup; recovery and route chunks stay deferred; cache population uses at most three workers.

## Release gates

Frontend tests cover:

- root-save versus drawer-clear exclusion in both directions, exact blocker identity and lazy remount continuity;
- one real retry request, truthful exhaustion UI and Feature/Skills Runtime recovery isolation;
- current-shell offline ordering, previous hash-chunk continuity, current-only metadata, exact query matching and awaited navigation writes;
- Save-Data/2G suppression and recovery-layer deferral.

`python scripts/smoke_frontend_browser.py` records the corresponding Chromium Evidence, while `python scripts/check_frontend_bundle.py` verifies the layered manifest without changing the 4.3.0 bundle budgets.

## Compatibility and non-goals

- No new Provider split, backend protocol, dependency upgrade, ChatContext migration or offline Mutation persistence.
- No Project/Skill behavior change, drawer redesign, HistoryDrawer loading change or JS/CSS budget increase.
- The frozen 4.0 runtime contract, Python ownership, opt-in Rust delegates and fallback guarantees remain unchanged.
- Formal GA Evidence remains bound to the clean exact merge by the 4.2.8 assembly and package verification chain.
