## Mutation Intent and Remount Continuity

4.2.5 is the final focused reliability patch in the 4.2.x Workspace Mutation series. It changes no backend protocol and adds no dependency or architecture surface.

### What changed

- Lifecycle locks compare both operation and normalized intent. Only genuinely identical payloads share the original Promise; mismatched rename, upload, Skill and toggle requests return an explicit conflict.
- Project, Skill, Memory and project-binding writes record owner, entity, operation and intent metadata in MutationCache.
- Project/Skill creation uses the same intent locks as entity writes, closing same-frame and remount duplicate-submission windows.
- Coordination conflicts become controller-owned user-visible errors with active-operation wording; recovery and subsequent successful work clear them.
- Pending and paused create, upload, clear and binding state is derived from MutationCache and therefore survives Workspace Provider remounts.
- Memory save intent normalizes whitespace, defaults and replacement-ID order. The Memory clear barrier also inspects cache-resident writes.
- Project-binding save state and the latest failure survive remount; retry replays the latest failed variables.

### Verification

- Hook contracts cover all 14 required intent, conflict, duplicate-create and remount-continuity cases.
