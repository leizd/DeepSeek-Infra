## Evidence Provenance and Blocker Attribution

4.2.7 closes two trust gaps without extending the Workspace state architecture or changing the backend protocol.

### What changed

- Every Project, Skill, Memory and project-binding lifecycle Mutation has a stable `lifecycleId`. Conflict results identify the requested action separately from the exact blocker entity, operation, intent and source.
- Coordination messages follow the blocker's lifecycle ID. Cross-entity Memory barriers remain visible while the real clear/save/remove task is active, then disappear even if another Mutation starts on the same entity.
- A project-binding save blocks deletion of that project, and project deletion blocks binding edits. Rename and upload remain independent from binding changes.
- `scripts/generate_release_evidence.py` requires a clean, version-synchronized release candidate, captures its HEAD once, and shares that immutable context across all generated Evidence.
- `--provenance-strict` verifies clean source identity, known tested revision, CI `GITHUB_SHA`, required Evidence membership and per-file SHA-256 checksums.

### Verification

The frontend contracts cover both Memory barrier directions, cross-entity error persistence, exact blocker settlement, same-entity replacement, and project/binding coordination. Chromium Evidence adds:

```text
crossEntityBlockerAttributed
crossEntityConflictPersists
exactBlockerSettlementClears
projectBindingBlocksDeletion
projectDeletionBlocksBinding
```

Release-candidate workflow:

```text
# 1. Commit the source candidate and confirm a clean worktree.
python scripts/generate_release_evidence.py --version 4.2.7

# 2. Review and commit the generated Evidence and release notes.
# CI regenerates final Artifact Evidence with ciRevision == GITHUB_SHA.
```

Local candidate Evidence intentionally leaves `releaseRevision` and `ciRevision` null. It proves the clean candidate source; CI Artifact Evidence proves the final release HEAD without inventing a self-referential commit SHA.

### Compatibility

Python remains authoritative, Rust delegation remains opt-in with Python fallback, and the frozen 4.0 protocol contract is unchanged. No dependency, Provider lifecycle, Query persistence, lazy loading or bundle-budget change is included.
