## Mutation Completion Reconciliation

4.2.6 completes the focused 4.2.x Workspace Mutation reliability series. It changes no backend protocol and adds no dependency or architecture surface.

### What changed

- Project creation captures the active-selection revision and activates the new project only if the user did not select another project while waiting. Project deletion checks the live selection at completion instead of a request-time closure.
- Project create and rename completion handlers compare the submitted target and normalized draft before clearing inputs or closing editors.
- Skill forms use submission generations and normalized draft intents. Editing, canceling, closing, or reopening a form invalidates an older completion callback.
- Pending Memory suggestions carry stable IDs. Save success and conflict results update only the suggestion that initiated the request.
- Coordination conflicts retain entity and operation identity, automatically disappearing once the blocking lifecycle Mutation is no longer active. Binding retry/refetch clears stale coordination state explicitly.
- Project, Skill, and Memory errors are reconciled by entity: another entity's success cannot hide a failure, while a later success on the same entity resolves its old failure.

### Verification

- Hook and component contracts cover active-project selection, Project and Skill drafts, Memory suggestion identity, coordination expiry, binding recovery, and per-entity failures.
- Chromium evidence adds `projectSelectionLatestIntentWins`, `workspaceDraftLatestIntentWins`, `renameCompletionIsolation`, `skillFormCompletionIsolation`, `memorySuggestionCompletionIsolation`, `coordinationErrorAutoClears`, `bindingCoordinationRecovery`, and `lateConcurrentFailureVisible`.
- The initial frontend entry remains below the unchanged 450 KB budget.
