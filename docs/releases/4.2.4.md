## Mutation Scope and Lifecycle Integrity

4.2.4 is a focused Workspace Mutation integrity patch. It changes no backend protocol and adds no new architecture surface.

### What changed

- Project-list, Project-binding, Skill-list and Memory-list mutations use separate, exactly matched ownership keys.
- Controller recovery removes only failed owned mutations; pending, paused and successful work stays visible in MutationCache and UI state.
- Project rename/remove/upload and Skill update/toggle/remove coordinate through entity lifecycle locks. Identical operations deduplicate; conflicting operations do not dispatch.
- Project uploads capture the initiating project ID in their variables. Switching projects no longer moves the upload indicator or changes its target.
- Memory save is represented as a Mutation. Expected conflict responses remain normal business outcomes, while clear is exclusive against active save/remove work.

### Verification

- Hook contracts cover binding isolation, pending recovery, Project/Skill conflicts, stable upload targeting and both directions of the Memory write barrier.
- Chromium evidence adds `mutationScopeIsolation`, `bindingErrorRemainsLocal`, `recoveryPreservesPendingWork`, `projectLifecycleActionsExclusive`, `skillLifecycleActionsExclusive`, `projectUploadTargetStable`, `projectUploadBlocksDeletion` and `memoryClearWriteBarrier`.
- The initial frontend entry remains below the unchanged 450 KB budget.

### Compatibility

Python remains authoritative, Rust delegation remains opt-in with Python fallback, and the frozen 4.0 protocol contract is unchanged.
