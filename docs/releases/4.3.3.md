# DeepSeek Infra 4.3.3 — Update Discovery and Quiescent Reload

<!-- docs-language-switcher:start -->
[中文](../../README.md) / [English](../../README.en.md)
<!-- docs-language-switcher:end -->

4.3.3 closes the gap between immutable frontend builds and safe adoption of a newly deployed build. The application can discover and stage an update without interrupting active work, and it reloads only after explicit consent and verified quiescence.

## Deployment discovery

- The stable `/ui/workspace-assets.json` pointer is fetched with `no-store` at startup, whenever a visible tab returns online or visible, every five minutes while visible, and through the manual check action.
- Pointer payloads must match schema 1 and valid version, source revision, 16-character build ID, and 64-character asset digest fields.
- Concurrent checks share one request. Aborted, invalid, failed, or stale requests cannot announce an update, and a newer C target supersedes every pending B result.
- `BroadcastChannel` shares build-available, staged, and activated facts across tabs without issuing a peer-tab reload.

## Staged Worker activation

- A first Service Worker install still calls `skipWaiting()` after its Core Cache is ready.
- An upgrade installs and remains waiting. It accepts activation only from an exact `activate_build` message whose `buildId` and `assetSetDigest` match its embedded identity and whose shell plus immutable manifest are cached.
- The page verifies the waiting Worker before consent, then verifies the controlling Worker again after `controllerchange`.
- A reload is rejected on timeout, identity/digest/cache mismatch, a superseded target, a new blocker, or a failed synchronous persistence flush.

## Quiescent reload and drafts

- The external build-update store exposes `current`, `checking`, `available`, `installing`, `ready`, `blocked`, `activating`, `reload-required`, and `error` phases through `useSyncExternalStore`.
- The compact update banner supports retry, defer, explicit reload, and “complete current work, then update” behavior.
- Reload blockers cover chat/Agent streaming, every pending React Query mutation, attachment upload, ready unsent attachments, quote drafts, Composer persistence, and Skills, Projects, and Reminders forms.
- Composer text is stored per conversation in `sessionStorage` after a 120 ms debounce. Files and credentials are never persisted. Composer and conversation state expose synchronous flush paths used before activation.

## HTTP cache contract

- `index.html`, SPA HTML fallbacks, and `/ui/workspace-assets.json` use `Cache-Control: no-store`.
- `workspace-assets-<buildId>.json`, `sw-<buildId>.js`, `sw-root-<buildId>.js`, and hashed Vite assets use `public, max-age=31536000, immutable`.
- Generic static/vendor resources retain the existing `no-cache` policy.
