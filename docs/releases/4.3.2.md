# DeepSeek Infra 4.3.2 — Immutable Build Identity and Multi-Tab Cache Handoff

<!-- docs-language-switcher:start -->
[中文](../../README.md) / [English](../../README.en.md)
<!-- docs-language-switcher:end -->

4.3.2 binds each page, Service Worker, asset manifest and Cache to one immutable frontend build. It also protects long-lived tabs across repeated deployments without warming the wrong build or deleting leased lazy chunks.

## Build identity

- `buildId` is derived before bundling from the frontend version, exact source revision and a versioned build-configuration contract.
- Formal CI uses `GITHUB_SHA`; clean local builds use `git rev-parse HEAD`; dirty frontend builds append a deterministic source digest.
- The build emits `workspace-assets-<buildId>.json`, `sw-<buildId>.js` and `sw-root-<buildId>.js`. Each worker embeds its own build and digest and refuses a mismatched immutable manifest.
- `assetSetDigest` independently covers the emitted page and offline assets, canonical manifest content and Service Worker templates.

## Controller handshake and warmup

- The page registers its build-scoped worker with `updateViaCache: "none"`.
- Warmup uses the controlling worker, never merely `registration.active`, and starts only after a MessageChannel response matches the page build.
- Concurrent tabs share one worker warmup task. Exact cache hits are skipped, partial failures leave no completion marker, and the next attempt requests only missing assets.
- Save-Data, slow-2g and 2g suppression remain unchanged; Recovery and route-optional chunks remain outside primary warmup.

## Multi-build Cache leases

- Pages report their own build on initial control, `controllerchange`, visible-state return and a 60-second heartbeat.
- Workers retain the current build, the immediately previous build and every build leased by an active or recently observed client.
- Activation claims clients and requests leases before pruning. Closed clients become removable only after lease expiry and reconciliation with `clients.matchAll`.
- Exact old hash chunks remain readable by leased pages through A→B→C deployments; unhashed metadata and query-altered requests never cross build boundaries.
