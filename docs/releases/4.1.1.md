## Trace Loading Resilience

4.1.1 hardens the routed and Diagnostics Trace experience introduced in 4.1.0. A failed Trace request now exposes an in-place Retry action. Retrying replaces the active request with a fresh `AbortController` and returns the view to its loading state without reloading the route or mounting Workspace providers.

Trace state updates now check the request signal on both success and failure. Late resolutions are ignored after a route change, retry, or unmount even when an injected or custom HTTP client does not honor cancellation. The normal browser client still aborts the underlying `fetch` through the shared `HttpClient`.

Trace recovery UI and styles remain owned by the lazy Trace detail chunk. This patch does not introduce TanStack Query, rewrite Context providers, or change the frozen 4.0 backend protocol and Python-first runtime boundary.

## Executable contracts

React component tests cover initial failure, retry success, replacement-request cancellation, unmount cancellation, and stale-response suppression. The Chromium smoke intercepts a dedicated Trace request, returns HTTP 503 on the first attempt, succeeds on the second, and verifies recovery through the visible Retry action.

Starting with 4.1.1, preflight requires the `traceRetryRecovery` PASS check in versioned frontend browser evidence. Existing deferred-chunk, provider-isolation, navigation, refresh, offline, CSP, and bundle contracts remain required.
