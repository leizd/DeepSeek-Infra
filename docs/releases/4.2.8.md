## Exact-Merge Evidence Assembly

4.2.8 closes the 4.2.x line by making release Evidence a single auditable exact-merge product. It does not extend Workspace Mutation behavior or change frontend dependencies, Provider ownership, Query persistence, lazy loading, offline persistence, bundle budgets, backend protocols or product features.

### What changed

- A dedicated `evidence-context` job captures one clean schema-v2 context containing version, tested revision, repository, workflow run and attempt, event and ref identity.
- Every Evidence producer receives that context and writes provenance directly. The release generator no longer edits reports after they are produced.
- One typed `EvidenceSpec` inventory defines each path, owning producer, candidate/exact-merge tier and GA requirement for generation, preflight, manifest construction, assembly and tests.
- Producer outputs are uploaded in isolated directories. Final assembly rejects missing reports, extra ownership, path collisions, stale versions, non-PASS state and revision or CI identity drift.
- The assembled manifest has a detached SHA-256. The release manifest records its path, checksum, entry count and tested revision.
- CI builds the real release ZIP, restores the assembled Evidence into it, and verifies byte identity, checksums, revision binding, secret/runtime exclusions and the absence of unlisted current-version Evidence.

### CI chain

```text
evidence-context
  -> isolated candidate and exact-merge producer Artifacts
  -> evidence-assembly
  -> release-evidence-v4.2.8
  -> release-package
  -> verified ZIP + checksum + release manifest + final preflight
```

Candidate-tier reports can be exercised before merge, but GA still requires every exact-merge producer and the final package verifier to pass against the same `GITHUB_SHA`. Formal 4.2.8 Evidence is therefore generated by CI after the source candidate is committed, never synthesized from a dirty tree.

### Compatibility

The 4.2.7 frontend lifecycle behavior remains intact. Python stays authoritative, Rust delegation stays opt-in with guaranteed Python fallback, and the frozen 4.0 protocol contract is unchanged.
